# LLM과 RAG

## LLM의 한계점
▪️ 학습 마감일 문제 (지식 컷오프 문제)</br>
    - LLM은 특정 시점까지의 데이터로 학습됨. '지식 컷오프 (knowledge Cutoff)' 라고 함.</br>
▪️ 환각 (Hallucination)</br>
    - 모르는 정보를 마치 아는 듯 대답하는 현상</br>
    - LLM은 확률적으로 가장 그럴듯한 다음단어를 예측하는 방식이기 때문.</br>
▪️ 개인화 불가</br>
    - 일반적인 특성에 대한 대답을 잘 구사하지만, 사내자료, 도메인 적인 지식 등에 대한 대답은 불가함.</br>
</br>

## RAG 란
▪️ Retrieval-Augmented Generation '검색 증강 생성'</br>
    - Retrieval(검색) : 정보를 먼저 찾고</br>
    - Augmented(증강) : 그 정보로 LLM을 보강하고</br>
    - Generation(생성) : 답변을 생성함</br>
-> 즉, RAG는 검색하고 이를 기반으로 답변을 생성함</br>
▪️ RAG의 장점</br>
    - 최신 정보 제공</br>
        * llm 을 다시 학습시키지 않아도, 지식 저장소만 업데이트하면 최신 지식 기반으로 답변이 가능</br>
    - 환각 감소 </br>
        * 실제 문서를 기반으로 답변하므로 없는 내용을 지어낼 확률이 감소</br>
    - 출처 제공 가능</br>
        * llm 이 어떤 문서를 참고해서 답변했는지 명확하게 답할 수 있음 </br>
    - 맞춤형 AI 가능
</br>

### RAG 의 구성요소
#### 1️⃣ 지식 저장소 (Knowledge Base)
검색할 문서가 저장된 곳</br>
- 회사 내부문서</br>
- 제품 매뉴얼</br>
- 설계 공정에 대한 매뉴얼</br>
- 기술 문서 등</br>
</br>

#### 2️⃣ 검색 시스템 (Retriever)
질문과 관련된 문서를 찾아줌</br>
'''
질문: 컴퓨터 초기설정 방법 알려줘
↓
검색시스템: 지식 저장소에서 관련 문서 검색
↓
결과: "회사 내부 문서 p.1"
'''
</br>

### 3️⃣ 생성모델 (generator)
LLM이 답변 생성</br>
'''
검색된 문서 + 사용자 질문
↓
LLM: 문서 내용을 읽고 이해하고 질문에 맞게 답변 생성
↓
"xxx.com url로 진입하여 필요한 패키지를 설치하고, 초기 비밀번호는 ㅇㅇㅇ입니다.~ "
'''
</br>

## GraphRAG
### 기존 RAG의 한계
#### 1️⃣ 점 검색의 문제
▪️ 기존의 RAG는 문서를 작은 청크로 나누고, 질문과 가장 유사한 청크를 찾아서 답변함. </br>
'''
[문서 A - 직원 정보]
"김철수는 개발팀 소속이다."

[문서 B - 팀 정보]  
"개발팀의 팀장은 박영희이다."

[문서 C - 프로젝트 정보]
"박영희는 AI 프로젝트를 담당한다."

'''
</br>
-> 질문: "김철수의 팀장은 누구인가요?"</br>
₩₩₩
전통 RAG의 처리:

1. "김철수"와 "팀장"이 포함된 문서 검색
2. 문서 A만 검색됨 (김철수 언급)
3. 문서 A에는 팀장 정보가 없음
4. 답변 실패 또는 부정확한 답변
